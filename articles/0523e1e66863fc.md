# はじめに

[cuda-oxide](https://github.com/NVlabs/cuda-oxide)が公開されて、色々見ていたところ、そこで定義されていた`DisjointSlice`が気になったので調べてみました。


# cuda-oxideとは

[cuda-oxide](https://github.com/NVlabs/cuda-oxide)はRustからCUDAに変換するコンパイラです。

https://github.com/NVlabs/cuda-oxide/blob/faea3959950d3bb40efb6a29d03bfbc65725bae6/crates/rustc-codegen-cuda/examples/vecadd/src/main.rs#L35-L47

こんな感じでRustでCUDAカーネルを書くことができます。この例では`a`, `b`の配列を受け取り、各要素を足して`c`に入れています。

## CUDAカーネルは並列に実行される

CUDAに馴染みのない方向けに、少しだけ補足します。

普通のRustの関数は、書いたら一回の呼び出しで一度だけ実行されます。しかしCUDAカーネルはそうではなく、**GPU上で何千・何万ものスレッドによって一斉に並列実行される**のが大きな特徴です。`vecadd`を呼び出すと、1つのスレッドが配列全体をループで処理するのではなく、たくさんのスレッドがそれぞれ「自分の担当する1要素」だけを計算します。

各スレッドは自分が何番目のスレッドなのかを知る手段を持っていて、`let idx = thread::index_1d();`（CUDAでいう`blockDim.x * blockIdx.x + threadIdx.x`）で「自分の番号」を受け取ります。そしてその番号を配列のインデックスとして使うことで、

- スレッド0は`c[0] = a[0] + b[0]`
- スレッド1は`c[1] = a[1] + b[1]`
- スレッド2は`c[2] = a[2] + b[2]`
- ……

というように、各スレッドが配列の異なる位置を**同時に**書き込みます。

`cuda-oxide`はRustのMIRからMLIRライクな[Pliron](https://github.com/pliron-org/pliron)をつかってLLVM IRにしてPTX(CUDAが受け取る形式)にコンパイルします。

![pipeline](https://nvlabs.github.io/cuda-oxide/_images/compiler-pipeline.svg)

かなりよく動いていて、Rustのほぼ全ての機能をそのままカーネルに持ち込めるのがありがたいです。

# CUDAとRustのSafety

先ほど`cuda-oxide`から引用した`vecadd`ですが、これをCUDAで書くと大体こんな感じです。

```cpp
__global__ void vecadd(int N, const float *a, const float *b, float *c)
{
    int i = blockDim.x * blockIdx.x + threadIdx.x;
    if(i < N) {
        c[i] = a[i] + b[i];
    }
}
```

これをRust視点でよく見てみると、`a`, `b`は読み込みだけなのでスレッド間で共有していいですが、`c`は書き込み可能です。あらゆるスレッドが好きに`c`へ書き込めてしまうと、明確に **Shared XOR Mutable** に反してしまいます。
これを安全に解決するのが`DisjointSlice`です。

# DisjointSliceの使い方

もういちど

https://github.com/NVlabs/cuda-oxide/blob/faea3959950d3bb40efb6a29d03bfbc65725bae6/crates/rustc-codegen-cuda/examples/vecadd/src/main.rs#L35-L47

を見てみると`c`は`DisjointSlice`型で、`let idx = thread::index_1d();`でインデックスを得てから`c.get_mut(idx)`で`c`にアクセスしています。
セーフRustでは`DisjointSlice`に対して好きなインデックスを作ることはできず、必ずあらかじめ認められている方法でしかインデックスを作ることができません。
この例で言うと、`thread::index_1d`は必ずユニークなインデックスを返すので`c`にアクセスしても安全というわけです。

## ThreadIndexはスレッド間で共有してはいけない

CUDAにはShared Memoryというメモリがあります。このメモリは同じブロック内の複数スレッドで共有できます。

`cuda-oxide`では`static`変数を使用するとShared Memoryを使えます。

https://github.com/NVlabs/cuda-oxide/blob/a6b70f0e363e2d6bb9e28ffd1e1eeac4455beafc/crates/rustc-codegen-cuda/examples/sharedmem/src/main.rs#L31-L51

Shared Memory経由でインデックスをやり取りできてしまうと、まったく安全ではなくなります。そのため、ライフタイムによってそれを制限しています。

試しに、`static`なShared Memoryに`ThreadIndex`を保管して別スレッドへ横流ししようとすると、コンパイルが通りません。

```rust
use cuda_device::{device, SharedArray};
use cuda_device::thread::{self, ThreadIndex};

// ThreadIndex を Shared Memory に置こうとする（'static が要求される）
static mut COURIER: SharedArray<ThreadIndex<'static>, 1> = SharedArray::UNINIT;

#[device]
pub fn bad_shared_laundering() {
    unsafe {
        COURIER[0] = thread::index_1d(); // ← ここで弾かれる
    }
    let _stolen = unsafe { COURIER[0] };
}
```

```
error[E0716]: temporary value dropped while borrowed
  --> tests/compile_fail/thread_index_shared_laundering.rs:11:1
   |
11 | #[device]
   | ^^^^^^^^^
   | |
   | creates a temporary value which is freed while still in use
   | assignment requires that borrow lasts for `'static`
...
18 | }
   | - temporary value is freed at the end of this statement
```

`static`に置くには`ThreadIndex<'static>`が必要ですが、`index_1d()`が返す`ThreadIndex`の`'kernel`はカーネル本体に紐づくローカルなライフタイムなので`'static`へは伸ばせず、コンパイルエラーになります。これによってインデックスをShared Memory経由で他スレッドへ渡すことが、型レベルで防がれています。

さらに`ThreadIndex`は`Copy`も実装していないため、仮に格納できたとしても取り出し側が`error[E0507]: cannot move out of index of ...`で弾かれ、二重の防御になっています。

> この例は cuda-oxide のテストスイート（`crates/cuda-device/tests/compile_fail/thread_index_shared_laundering.rs`）にコンパイル失敗テストとして含まれており、上記のエラーはその期待出力です。


# `DisjointSlice`の中身

`DisjointSlice`が実質的に持っているのはポインタと長さだけで、`ThreadIndex`はインデックスを表す整数のみ持っています。
安全性はゼロコストで実装されています。

```rust
#[repr(C)]
pub struct DisjointSlice<'a, T, IndexSpace = Index1D> {
    ptr: *mut T,
    len: usize,
    _marker: PhantomData<&'a mut [T]>,
    _space: PhantomData<fn() -> IndexSpace>,
}

impl<'a, T, IndexSpace> DisjointSlice<'a, T, IndexSpace> {
    /*
        省略
    */
    #[inline]
    pub fn get_mut<'kernel>(&mut self, idx: ThreadIndex<'kernel, IndexSpace>) -> Option<&mut T> {
        let i = idx.get();
        if i < self.len {
            // SAFETY:
            // - Bounds check passed above.
            // - `idx` is a ThreadIndex derived from hardware built-in variables,
            //   guaranteeing a unique index per thread (no data races).
            // - The DisjointSlice was constructed with valid memory (from_raw_parts safety).
            Some(unsafe { &mut *self.ptr.add(i) })
        } else {
            None
        }
    }
    /*
        省略
    */
}
```

```rust
pub struct ThreadIndex<'kernel, IndexSpace = Index1D> {
    raw: usize,
    _kernel: PhantomData<fn(&'kernel mut ()) -> &'kernel mut ()>,
    _space: PhantomData<fn() -> IndexSpace>,
    _not_send_sync: PhantomData<*mut ()>,
}

impl<'kernel, IndexSpace> ThreadIndex<'kernel, IndexSpace> {
    /*
        省略
    */
    #[inline(always)]
    unsafe fn new(raw: usize, _scope: &'kernel KernelScope<'kernel>) -> Self {
        ThreadIndex {
            raw,
            _kernel: PhantomData,
            _space: PhantomData,
            _not_send_sync: PhantomData,
        }
    }
}
```

`&'kernel`はカーネルのスコープ全体の範囲のライフタイムで、`ThreadIndex`ではマーカーでその範囲を伸ばすことも短くすることもできないようにしています。
動かすにはまず`&'kernel`をどこかで作らなければならないので、`#[kernel]`proc-macro(`cuda-macros`クレート)が、カーネル内で`thread::index_1d`などのインデックス呼び出しを見つけた場合に、ライフタイムを持ったマーカー(`KernelScope`)を生成するコードを本体の先頭に自動挿入します。

https://github.com/NVlabs/cuda-oxide/blob/c3caeb8ea32af4b622054b7591275649f13d45b1/crates/cuda-macros/src/lib.rs#L1690-L1704

同時にマクロは`thread::index_1d()`などの呼び出し側を書き換えてこのマーカーを引数に差し込み、その先で`ThreadIndex::new`にマーカーが渡されます。

# マーカー

`DisjointSlice`, `ThreadIndex`にあるマーカーを一つずつ見ていきます。

```rust
#[repr(C)]
pub struct DisjointSlice<'a, T, IndexSpace = Index1D> {
    ptr: *mut T,
    len: usize,
    _marker: PhantomData<&'a mut [T]>, // 'a をここで使う。 mutなのでinvariant
    _space: PhantomData<fn() -> IndexSpace>, // ここでIndexSpaceを使う。`fn() -> IndexSpace`は型を所有せず使うことでIndexSpace由来の性質を持ち込まないためのイディオム
}
```

```rust
pub struct ThreadIndex<'kernel, IndexSpace = Index1D> {
    raw: usize,
    _kernel: PhantomData<fn(&'kernel mut ()) -> &'kernel mut ()>, // 'kernel をここで使う。 `fn(&'kernel mut ()) -> &'kernel mut ()`は&'kernelを伸ばすことも短くすることもできないように固定(invariant)するイディオム
    _space: PhantomData<fn() -> IndexSpace>, // IndexSpace使用
    _not_send_sync: PhantomData<*mut ()>, // !Send + !Syncを型に入れるイディオム。ThreadIndexはthreadごとに固有のはずなので!Send + !Syncを型に入れる必要がある。
}
```

# `index_1d`は安全か？

https://github.com/NVlabs/cuda-oxide/blob/a674faeab86bab7b161a009ec3b474edf51a3e66/crates/cuda-device/src/thread.rs#L280-L288

`thread::index_1d()`はグリッドのx次元しか見ていないので例えばカーネル呼び出し時に`blockDim_y` > 1だとyだけ違うブロックが同じインデックスを返します。

もし呼び出し時にブロックもしくはグリッドのy,z次元が1でないとインデックスが衝突してしまいます。
カーネル呼び出しはセーフなので全部セーフなのにメモリの衝突が起きてしまうのでよくなさそうです。

なので
https://github.com/NVlabs/cuda-oxide/blob/faea3959950d3bb40efb6a29d03bfbc65725bae6/crates/cuda-device/src/thread.rs#L30-L31

というドキュメントは少し強すぎる気がします。

興味のある方は

https://github.com/NVlabs/cuda-oxide/issues/115

を読んでみてください。


# 感想

proc-macroを使ってマーカーを先頭に挿入してまで安全性を担保するのはかなり野心的だと思いました。
しかし書き込みのパターンがあらかじめ決められた仕組みに載っていないと使えないので、現実的には普通にunsafeでアクセスする機会の方が多くなるような気がしました。
とはいえ、[cute](https://docs.nvidia.com/cutlass/latest/media/docs/cpp/cute/00_quickstart.html)のようにジェネリクスで色々頑張ればかなりのアクセスパターンに対応できるようになる気もします。
