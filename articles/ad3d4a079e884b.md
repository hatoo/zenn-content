# はじめに

rustc 1.84.0から内部で使われるハッシュ関数が新しくなっていたので調べてみました。
ちなみに以前のハッシュ関数、FxHashについては[こちら](https://qiita.com/hatoo@github/items/1c627987991a0156d26c)に記事を書いているので興味があればご覧ください

後方互換性のために新しいハッシュ関数もFxHashの名前で公開されていて、特に新しい名前もないようです
この記事では以前のハッシュ関数をFxHash、新しい方を「新しいハッシュ関数」を呼ぶことにします

# FxHashの問題点

[rust-lang/rustc-hash#37](https://github.com/rust-lang/rustc-hash/pull/37)でFxHashの問題点で指摘され、新しいハッシュ関数が提起されました

FxHashのコードの重要な部分を引用します

https://github.com/rust-lang/rustc-hash/blob/0773e83fddff56670e107134dbd9f12e6b6ecdf4/src/lib.rs#L51-L140

問題点を要約すると、

## 1. 長いバイト列をハッシュ化するのが遅い

別に特別悪いわけではないと思うが、たしかにどうせ入力が長いなら律儀に全bit同じように処理しなくてある程度雑に処理して良い気がします

## 2. `finish()`が何もしない

最終的にハッシュ値を得るために`finish()`が呼ばれるのですがそこで何もしないので入力された最後の整数(もしくは単純に入力長がsize_of::<usize>()以下)のxor後に左回転が行われない
ということは実質xorして掛け算するだけなので線形合同法を同じようにその入力が貢献するLSB付近への影響が悪い
例えばハッシュテーブルのバケットにLSBから順にハッシュ値を使うことが多いので問題になる

# rustc-hash 2の実装

https://github.com/rust-lang/rustc-hash/blob/dc5c33f1283de2da64d8d7a06401d91aded03ad4/src/lib.rs#L51-L191


まずメインの処理が

```rust
impl FxHasher {
    #[inline]
    fn add_to_hash(&mut self, i: usize) {
        self.hash = self.hash.wrapping_add(i).wrapping_mul(K);
    }
}
```

もう足して掛け算してるだけです。

https://github.com/rust-lang/rustc-hash/blob/dc5c33f1283de2da64d8d7a06401d91aded03ad4/src/lib.rs#L63-L75

ここのコメントにあるようにこれは各入力にMCG^[線形合同法(LCG)から足し算の項を省いたやつ]で作られた乱数を掛けて足しているように解釈できるのでKもMCGの定数から持ってきています。

```rust
    #[inline]
    fn finish(&self) -> u64 {
        // Since we used a multiplicative hash our top bits have the most
        // entropy (with the top bit having the most, decreasing as you go).
        // As most hash table implementations (including hashbrown) compute
        // the bucket index from the bottom bits we want to move bits from the
        // top to the bottom. Ideally we'd rotate left by exactly the hash table
        // size, but as we don't know this we'll choose 26 bits, giving decent
        // entropy up until 2^26 table sizes. On 32-bit hosts we'll dial it
        // back down a bit to 15 bits.

        #[cfg(target_pointer_width = "64")]
        const ROTATE: u32 = 26;
        #[cfg(target_pointer_width = "32")]
        const ROTATE: u32 = 15;

        self.hash.rotate_left(ROTATE) as u64

        // A bit reversal would be even better, except hashbrown also expects
        // good entropy in the top 7 bits and a bit reverse would fill those
        // bits with low entropy. More importantly, bit reversals are very slow
        // on x86-64. A byte reversal is relatively fast, but still has a 2
        // cycle latency on x86-64 compared to the 1 cycle latency of a rotate.
        // It also suffers from the hashbrown-top-7-bit-issue.
    }
```

で`finish()`で左回転して質の悪いLSB付近を真ん中に寄せています

```rust
    fn write(&mut self, bytes: &[u8]) {
        // Compress the byte string to a single u64 and add to our hash.
        self.write_u64(hash_bytes(bytes));
    }
```

では`bytes`を[wyhash](https://github.com/wangyi-fudan/yhash)からインスパイアされた`hash_bytes`で64bitにまとめてから処理します
`hash_bytes`自体はMCGの要素は使っておらず、速度に特化した雑なハッシュ関数といった印象です

# rustc-hash 2.1.1

[rust-lang/rust#135477](https://github.com/rust-lang/rust/issues/135477)

で新しいハッシュ関数(rustc-hash 2.0)を搭載したrustcでレグレッションが発生しました

https://github.com/rust-lang/rustc-hash/blob/8f258eec1f9e4a328fa0f7d370fcf7d51251ce96/src/lib.rs#L166-L190

2.0では64bit環境で20左回転していたのですが、そうなると21bit目から質の悪いLSB付近のビットが配置されてしまいサイズが2^20以上の巨大ハッシュテーブルで衝突が増大してしまったようです。設計者はrustcで2^20以上のハッシュテーブルが作られることはないと想定していたようです。

https://github.com/rust-lang/rustc-hash/pull/56

ということで2.1.1では26左回転するようになりました
